# Chambers Automotive Simulation Testbed

A multi-phase simulation testbed that validates the **Chambers sealed ephemeral computation model** for connected vehicles under realistic traffic conditions. The testbed wraps industry-standard simulators (SUMO вҶ’ CARLA вҶ’ ROS2/Gazebo) in a Chambers enforcement layer, producing measurable compliance artefacts and data-residue comparisons against a no-enforcement baseline.

---

## Background

Modern vehicles continuously generate high-bandwidth telemetry across five egress channels вҖ” cellular (4G/5G), Bluetooth, OBD-II, V2X (C-V2X / DSRC), and Wi-Fi вҖ” and distribute it to a heterogeneous ecosystem of stakeholders: OEM backend services, insurers, ADAS suppliers, Tier-1 component suppliers, and regulators. Current telematics architectures treat this as a fire-and-forget pipeline, with no per-session binding between the key that encrypts a drive and the policies that govern what survives it.

**Chambers** is an architecture that enforces data minimisation at the telematics enforcement point through three coupled mechanisms:

1. **Session Seal** вҖ” an ephemeral AES-256-GCM key is generated by a Hardware Security Module (HSM) at drive start and bound to that session only. The key never leaves the HSM boundary.
2. **Preservation Manifest** вҖ” a typed, stakeholder-scoped declaration of exactly what data categories survive the session, at what granularity, under which legal basis, for which jurisdictions.
3. **Burn Engine** вҖ” on drive end (ignition off / park), a 6-layer cryptographic erasure sequence destroys the session key and all undeclared residue, producing a signed destruction receipt.

This testbed exists to validate that architecture at scale: does the manifest evaluator correctly route data to declared stakeholders? Does the burn engine erase session state in full? Are the compliance artefacts sufficient for GDPR Art. 5(1)(e) and ePrivacy audit? Can consent be revoked mid-session without data leakage?

### Five Enforcement Points

Every Chambers gateway instance enforces the same manifest and burn semantics across all five vehicle egress channels simultaneously:

| EP | Channel | Typical stakeholders | Notes |
|----|---------|---------------------|-------|
| EP1 | Cellular (4G/5G) | OEM, insurer, ADAS | Primary high-bandwidth telemetry |
| EP2 | Bluetooth | Tier-1, workshop tools | Short-range diagnostic pairing |
| EP3 | OBD-II | Insurer (UBI), workshop | In-vehicle diagnostic bus |
| EP4 | V2X (C-V2X / DSRC) | Regulator, infrastructure | Cooperative awareness messages |
| EP5 | Wi-Fi | OEM OTA, fleet ops | Software update and fleet mgmt |

## Component Deep-Dives

### chambers-gateway (Rust)

Written in Rust 2021 with `tokio` async runtime. ~3,000 lines across seven modules.

**SoftHSM (`hsm/`)** вҖ” Simulates a PKCS#11 HSM using AES-256-GCM. Keys are stored in a thread-safe in-memory map behind an opaque `KeyHandle` (u64). Plaintext keys never appear in any external type; the API surfaces only encrypt/decrypt/destroy operations. On `destroy_key()`, the key bytes are zeroed before deallocation and a signed `DestructionReceipt` is returned for the audit log.

**Manifest Evaluator (`manifest/`)** вҖ” Loads a `PreservationManifest` (JSON) and, for each incoming `DataRecord`, returns a per-stakeholder list of `FilteredDataRecord` values. Filtering operates in three passes: (1) category match вҖ” is this `DataType` declared for this stakeholder? (2) field allow/deny вҖ” are any specific fields excluded? (3) granularity application вҖ” anonymise, aggregate, or score the record before forwarding. Consent revocation is tracked per-stakeholder per-session and immediately suppresses further routing to the revoked party.

**Burn Engine (`burn/`)** вҖ” Executes a strictly-ordered 6-layer erasure sequence when a session ends:

| Layer | Operation |
|-------|-----------|
| 1 | Logical deletion вҖ” mark all session blobs as deleted |
| 2 | Cryptographic erasure вҖ” destroy HSM key via `destroy_key()` |
| 3 | Storage overwrite вҖ” overwrite blob bytes with CSPRNG output |
| 4 | Memory zeroing вҖ” zero all in-process buffers |
| 5 | Semantic deletion вҖ” remove session-to-key mapping |
| 6 | Verification вҖ” confirm no session data remains accessible |

Each layer emits a `LayerResult` (duration, success flag, diagnostic detail). The aggregate `BurnReceipt` is signed and appended to the audit log.

**Audit Log (`audit/`)** вҖ” SQLite-backed, append-only event log. Each `AuditEntry` carries an HMAC-SHA256 over its payload concatenated with the previous entry's HMAC, forming a tamper-evident chain. Twelve `EventType` variants are tracked (SessionStart, DataGenerated, DataBlocked, BurnCompleted, ConsentRevoked, JurisdictionBlocked, PolicyViolation, etc.). The log supports chain verification, JSON export for regulators, and human-readable driver summaries.

**Gateway (`gateway/`)** вҖ” Wires HSM, manifest evaluator, burn engine, and audit log into a single `Gateway` struct. Entry points: `start_session()`, `process_record()`, `revoke_consent()`, `end_session()`. Exposes `GatewayMetrics` (total processed / transmitted / blocked / burned, active sessions). Falls back to a conservative no-telemetry default when the HSM is unavailable.

### chambers-sim (Python)

Python 3.11+, Pydantic v2, structlog, Click.

**SUMO Adapter** вҖ” Connects to SUMO via TraCI. Per-step, it reads each vehicle's speed, acceleration, and position, constructs typed `DataRecord` objects, and forwards them to either the local pure-Python gateway or the Rust gateway via HTTP. The adapter computes per-vehicle `driving_behaviour_score` (0вҖ“100) from speed and acceleration history and applies grid-snapping (`anonymise_position`) for anonymised position records.

**Local Gateway** вҖ” A pure-Python re-implementation of the Rust gateway's manifest evaluation and burn logic. Used in integration tests to keep tests hermetic (no subprocess, no network). Tracks the same session lifecycle and emits the same `ProcessingResult` / `BurnReceipt` types.

**Data Residue Analyzer** вҖ” Compares a baseline run (all data forwarded to all stakeholders, no filtering) against a Chambers-enabled run. Produces a `ResidueReport` with per-category and per-stakeholder byte and record reduction ratios, quantifying the privacy benefit of the enforcement layer.

### mock-stakeholders (FastAPI)

Four stateless endpoints that mirror the validation logic a real stakeholder backend would apply. Each endpoint rejects payloads that violate the declared manifest contract, providing a closed-loop test of whether the gateway correctly routed and filtered records:

- **`POST /oem/telemetry`** вҖ” accepts `sensor_health` only, at anonymised/aggregated granularity. Rejects raw GPS, raw driving behaviour, and camera frames.
- **`POST /insurer/trip`** вҖ” accepts `driving_behaviour` only at `per_trip_score` granularity. Rejects raw location and raw events.
- **`POST /adas/events`** вҖ” accepts `sealed_event` only. Rejects raw behaviour and contact metadata.
- **`POST /tier1/diagnostic`** вҖ” accepts `diagnostic_code` only.

## Preservation Manifest Format

Manifests are JSON documents validated against `manifests/schema.json` (JSON Schema v2020-12). A minimal example:

```json
{
  "manifest_version": "1.0",
  "vehicle_id": "VIN-DEMO-001",
  "session_id": "a1b2c3d4-...",
  "stakeholders": [
    {
      "id": "oem-backend",
      "role": "oem",
      "legal_basis": "contract",
      "categories": [
        {
          "type": "sensor_health",
          "granularity": "anonymised",
          "retention": "P90D",
          "purpose": "fleet reliability monitoring",
          "jurisdiction": ["EU"]
        }
      ]
    },
    {
      "id": "insurer-ubi",
      "role": "insurer",
      "legal_basis": "consent",
      "consent_ref": "UBI-CONSENT-2024-001",
      "categories": [
        {
          "type": "driving_behaviour",
          "granularity": "per_trip_score",
          "retention": "P365D",
          "purpose": "usage-based insurance pricing",
          "jurisdiction": ["EU", "UK"]
        }
      ]
    }
  ]
}
```

**Supported data types (14):** `sensor_health`, `driving_behaviour`, `sealed_event`, `gps_position`, `camera_frame`, `lidar_pointcloud`, `v2x_cam`, `bluetooth_pairing`, `wifi_passthrough`, `ecall`, `edr`, `security_event_log`, `diagnostic_code`, `adas_event`

**Supported granularities (7):** `raw`, `anonymised`, `aggregated`, `anonymised_aggregate`, `per_trip_score`, `pseudonymised`, `sealed`

**Legal bases (6):** `consent`, `contract`, `legal_obligation`, `vital_interest`, `public_interest`, `legitimate_interest` (GDPR Art. 6)

- **Rust** 1.78+ (`rustup toolchain install stable`)

## Quick Start

### Option A вҖ” Rust CLI demo (no simulator required)

```bash
cd chambers-gateway
cargo build --release
cargo run --release -- demo
```

The demo runs a synthetic 3-vehicle session, shows per-stakeholder filtering decisions, executes the 6-layer burn, verifies the audit chain, and prints a driver summary.

### Option B вҖ” Python demo (no simulator required)

```bash
cd chambers-sim
pip install -e ".[dev]"
chambers-sim demo --vehicles 10 --duration 60
```

### Option C вҖ” Full Phase 1 stack with Docker

```bash
docker compose -f docker/docker-compose.phase1.yml up --build
```

This starts four services: the Rust gateway, the FastAPI mock stakeholders (port 8000), the Python simulator adapter, and the SUMO process (TraCI port 8813).

## Building and Testing

```bash
# Rust gateway
cd chambers-gateway
cargo build --release
cargo test

# Python package
cd chambers-sim
pip install -e ".[dev]"
pytest

# Mock stakeholders
cd mock-stakeholders
pip install -e ".[dev]"
uvicorn app.main:app --reload
```

Integration tests in `tests/integration/` cover:
- Manifest enforcement (correct data type routing per stakeholder)
- Compliance artefacts (audit chain integrity, export format)
- Data residue (byte and record reduction vs. baseline)
- Consent revocation (mid-session cutoff, no trailing records)
- Threat simulation (16-threat model validation)
- Full session lifecycle (start вҶ’ process вҶ’ revoke вҶ’ burn вҶ’ verify)
- HSM fallback (conservative no-telemetry default)

## Conventions

- Rust: `thiserror` for structured errors, `tracing` for structured logging
- Python: Pydantic v2 models throughout, `structlog` for structured logging
- All manifests must validate against `manifests/schema.json`
- Tests are deterministic: no network calls, all external dependencies mocked
